1. Executive Summary
What is this product, who is it for, and why does it matter?

StarkPay is a consumer fintech mini-app modelled on the Opay experience. Users can send and receive tokens via their Privy wallet address, and every idle balance automatically earns yield through Vesu lending — so money is never sitting still.

The product is built entirely on the Starkzap SDK, which abstracts all blockchain complexity. Users interact with a familiar web2-style interface (email login, no seed phrases, no gas fees) while the app runs on Starknet L2 under the hood.

GOAL
Ship a production-grade, yield-bearing payment app in days, not months, using Starkzap's modular money toolkit.

1.1  Core Value Propositions
	•	Send & receive any supported token to any Privy wallet address instantly
	•	Idle balances auto-earn yield via Vesu (Starknet's lending protocol) — like a high-yield savings account built into your wallet
	•	Zero gas UX: AVNU paymaster absorbs all transaction fees
	•	Social login (email, Google, phone) — no wallet extensions, no seed phrases
	•	Fully non-custodial: Privy manages keys server-side; the user owns their wallet

2.2  User Flows
Send Money Flow
	•	User taps 'Send'
	•	Enters recipient Privy address (or scans QR)
	•	Enters token + amount
	•	SDK: withdraws required amount from Vesu lending position
	•	SDK: executes ERC20 transfer to recipient address
	•	AVNU paymaster covers gas fee — user sees $0 fee
	•	Confirmation screen shows transaction hash

Receive Money Flow
	•	User taps 'Receive'
	•	App displays Privy wallet address as QR code + copyable text
	•	Sender initiates transfer to that address
	•	Incoming balance auto-deposited into Vesu lending position
	•	User sees updated balance + new yield accrual

Yield Flow (background, automatic)
	•	User has idle USDC balance in wallet
	•	On deposit (or balance increase), SDK calls Vesu supply()
	•	Vesu allocates tokens to lending pool, APY begins accruing
	•	On next send, SDK calls Vesu withdraw() before ERC20 transfer
	•	Net UX: balance always shows total (principal + yield)

3.4  Wallet & Identity Model
Each user gets exactly one Privy embedded wallet on sign-up. This wallet address is their permanent identifier in the system — analogous to a phone number in Opay. There is no concept of a username-to-address lookup needed: the Privy address IS the user's payment address.

	•	One wallet per user, created on first login
	•	Privy stores keys on their secure servers (not on device, not on your backend)
	•	Users can export their private key later if desired (Privy provides this)
	•	No seed phrase shown during onboarding — fully web2-style UX

6. Key Implementation Details
The non-obvious engineering decisions and how to execute them.

6.1  SDK Initialisation
Starkzap is initialised once at app startup with your Privy app ID and AVNU paymaster key. The PrivySigner is created after the user logs in and passed to the SDK's WalletInterface.

CODE
import { StarkZap, PrivySigner } from 'starkzap';
const signer = new PrivySigner({ privyAppId: PRIVY_APP_ID, user: privyUser });
const zap = new StarkZap({ network: 'mainnet', signer, paymaster: { type: 'avnu', apiKey: AVNU_KEY } });

6.2  Yield Auto-Deposit on Receive
When a user receives tokens, the app listens for incoming transfer events. On detection, it calls Vesu supply() to move the new balance into the lending pool. This should be triggered client-side (or via a webhook) to keep the UX snappy.

	•	Listen: zap.token.onTransferReceived(callback)
	•	On trigger: zap.lending.supply({ token: 'USDC', amount: incomingAmount })
	•	Update UI: refetch Vesu position balance

6.3  Atomic Send with Yield Withdrawal
Sending money requires two on-chain steps: withdraw from Vesu, then transfer. Starkzap's transaction builder batches these into a single multicall, so the user only signs once and it appears atomic.

	•	Step 1: zap.lending.withdraw({ token: 'USDC', amount: sendAmount })
	•	Step 2: zap.token.transfer({ to: recipientAddress, token: 'USDC', amount: sendAmount })
	•	Batch both using zap.tx.batch([step1, step2]) — single signature, single txn

6.4  Balance Display
The displayed balance is the sum of the raw wallet balance plus the Vesu supply position. Always show these as a single combined number to the user.

FORMULA
displayBalance = walletBalance + vesuSupplyBalance + accruedYield

6.5  Gasless Transaction Setup
AVNU paymaster is configured at SDK init (see 6.1). No additional code is needed per transaction. The SDK wraps every transaction in the paymaster by default when the config is present.

You should set a daily gas budget limit in the AVNU dashboard to control costs. Consider building gas cost into a small spread or premium tier if you want to monetise.

9. Security & Compliance
Key risks and how the architecture mitigates them.

9.1  Security Considerations
	•	Private keys never touch your servers — Privy manages all key custody
	•	All transactions require Privy biometric or email confirmation — no silent signing
	•	AVNU paymaster key must be stored server-side only, never in the client bundle
	•	Rate limit all SDK calls on the backend to prevent draining the paymaster budget
	•	Input validation: always verify recipient addresses before constructing transactions
	•	Use Starknet testnet for all development; never use mainnet keys in development

9.2  Compliance Notes
	•	Privy's Terms of Service govern embedded wallet usage — review before launch
	•	Token transfers may be subject to local financial regulation depending on jurisdiction
	•	Consider KYC/AML integration (e.g. Persona, Onfido) for production if handling fiat on-ramps
	•	Vesu and AVNU are decentralised protocols — no direct compliance obligation, but document usage for legal review
